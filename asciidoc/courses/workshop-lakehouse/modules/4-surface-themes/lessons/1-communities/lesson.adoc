== Introduction

Your library now has cross-reference links - sections that point to other sections and documents.
Step back and look at the whole web: the connections are not evenly spread - they clump.

In this lesson, you will learn what community detection does and why running it over those cross-reference links produces meaningful themes.


[.slide]
== Communities are densely linked clusters

A **community** is a group of nodes more densely connected to each other than to the rest of the graph.

In the AutoFix library, the electrical manuals, the communication bulletins, and the wiring recalls all keep linking to the same diagnostic sections - the CAN-bus diagnosis procedure, the lost-communication writeups.
The brake documents converge on a different set of targets - pad-and-rotor service procedures and specs.
Almost nothing connects an electrical document to a brake document.
Those shared link targets pull documents together: two documents that both link to the same section are drawn into the same neighborhood, the strongest theme signal in any real corpus.

**Community detection** algorithms find those clusters automatically.
You will use **Leiden**, a Graph Data Science algorithm that assigns each node a community ID so that links inside communities are dense and links between them are sparse.


[.slide]
== Why these clusters mean something

Community detection on arbitrary data produces arbitrary clusters.
Yours will not, because of what each link _is_:

* Every edge exists because two documents **link to the same sections and documents** - or one explicitly links to the other
* A dense cluster of documents is therefore a set of documents about the **same repair topic**
* The cluster spanning the Falcon electrical manual, the BCM communication bulletin, and the wiring-harness recall is not a coincidence - it is the _electrical / CAN-bus theme_, assembled from documents that converge on the same diagnostic procedures

Nobody at AutoFix maintains a list of repair themes.
The themes exist anyway - as structure in the data.
That is the second shape: **patterns nobody named**.


[.slide]
== Why an agent needs themes

Without themes, an agent answering "where are we seeing repeat problems?" has to read everything - and blows past its context window.

With themes, the agent gets a small, structured view: a handful of evidence blocks, each with its member documents and the shared link targets that define it.
That fits in one prompt, and every claim in it traces back to real edges.

The spec for that view is `docs/theme-format.md` - read it before the challenge, the same way you read the outline spec.
