[.question]
= The Boundary Crossing

In the load pipeline, what joined the document half and the warehouse half into one graph?

- [ ] A. An entity resolution model matched document text to table rows
- [ ] B. A LINKS_TO relationship was created between every section and every work order
- [x] C. MERGE on part number and trouble code attached warehouse data to the same Part and DTC nodes the PDF sections already referenced
- [ ] D. The Delta tables were exported into the PDF parsing pipeline

[TIP,role=hint]
.Hint
====
Which `MERGE` key appeared in both `section_refs.csv` and the warehouse exports?
====

[TIP,role=solution]
.Solution
====
**C is correct:** `MERGE (p:Part {partNumber: ...})` and `MERGE (c:DTC {code: ...})` matched the nodes the PDF parsing created from section mentions, enriching them instead of duplicating them.
After that, one `Part` node is referenced by sections on one side and replaced on work orders on the other.

**Why others are wrong:**

* A: No model was needed - the identifiers were already shared; the lakehouse just never treated that as a link
* B: `LINKS_TO` connects sections to sections; the boundary crossing happens at the shared-key nodes
* D: The two halves stayed in their own pipelines - they meet in the graph

**Recall Module 4:** "The boundary crossing is one MERGE."
====
