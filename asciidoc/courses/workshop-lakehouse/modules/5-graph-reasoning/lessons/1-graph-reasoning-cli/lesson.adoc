== Introduction

Your tools are scripts your agent runs. But sometimes you - or the agent - just want to *ask the graph a question* without writing a tool.

**Advanced learners:** Skip to the finale.

**Everyone else:** This shows the `neo4j-cli` your agent already has, used directly for ad-hoc reasoning across both graphs.


[.slide]
== One command, any query

The Codespace ships `neo4j-cli` (the agent installed its skills in setup). It reads the same `.env`, so there is nothing to wire:

[source,shell]
.Ask the graph directly
----
neo4j-cli query "MATCH (s:Section)-[:LINKS_TO]->(tgt) RETURN coalesce(tgt.displayName, tgt.uri) AS target, count(*) AS links ORDER BY links DESC LIMIT 5"
----

This is what your agent runs under the hood when you ask it an open question. Try asking your agent _"which section or document is linked to by the most other sections?"_ and watch the `neo4j-cli` call it makes.


[.slide]
== Introspect the shape

When the agent meets an unfamiliar graph, the first move is to read its schema:

[source,shell]
.What's in here?
----
neo4j-cli query ":schema"
----

You will see both graphs at once: the document containment (`Library`, `Folder`, `Document`, `Section`) and the connections metadata (`Database`, `Schema`, `Table`, `Column`) that neocarta built. Two shapes, one database - the agent reads the schema to know which to traverse.


[.slide]
== Reason across both graphs

Because both graphs live in the same Neo4j, one query can span them.
A part number is not a graph node - it lives in section _text_ on the document side and as a `part_number` _column_ on the warehouse side.
So one query can find both at once: the warehouse tables that carry a `part_number` key, plus the library sections whose text mentions a given part string.

[source,shell]
.The document graph and the warehouse metadata, side by side
----
neo4j-cli query "MATCH (t:Table)-[:HAS_COLUMN]->(c:Column {name: 'part_number'}) RETURN t.name AS warehouseTable ORDER BY warehouseTable"
----

[source,shell]
.Find a part string in the library text (it is text, not a node)
----
neo4j-cli query "CALL db.index.fulltext.queryNodes('content_search', 'IC-2042') YIELD node, score RETURN node.uri AS section ORDER BY score DESC LIMIT 5"
----

Ad-hoc reasoning like this is the connective tissue between the shaped tools - the move an agent makes when a question does not fit a pre-built script.
