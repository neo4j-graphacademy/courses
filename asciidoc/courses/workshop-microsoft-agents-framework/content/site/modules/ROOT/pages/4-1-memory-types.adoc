= Memory Types

The context providers from Modules 2 and 3 inject pre-loaded knowledge — movie plots, genres, actors — from a graph that existed before the agent started. That knowledge is static. But agents also need to remember what happens _during_ conversations and carry that forward.

The link:https://github.com/neo4j-labs/agent-memory[neo4j-agent-memory^] library is a graph-native memory system for AI agents. It stores conversations, builds knowledge graphs from extracted entities, and lets agents learn from their own reasoning, all backed by Neo4j. The library provides framework integrations for LangChain, Pydantic AI, OpenAI Agents, and others. In this module, you will use its Microsoft Agent Framework integration, which wraps the library's memory capabilities into MAF context providers and callable tools.

== What Memory Needs to Capture

Three categories of information accumulate during agent conversations:

1. **What was said** -- the conversation itself. A user asking about time travel movies, the agent recommending Interstellar, the user saying they already saw it. This history needs to persist across turns and sessions so the agent doesn't repeat itself.

2. **What was learned** -- facts and preferences extracted from conversation. "The user prefers sci-fi" isn't a message; it's a structured piece of knowledge derived from messages. These extracted facts build up a profile of the user and the topics discussed.

3. **What was tried** -- the reasoning and tool calls the agent used to accomplish tasks. If the agent searched for time travel movies and found good results using a particular approach, that trace can inform how it handles similar requests later.

== Three Memory Types

The `neo4j-agent-memory` package maps these three categories to distinct memory types — short-term (conversation history), long-term (extracted knowledge), and reasoning (agent execution traces) — all backed by Neo4j.

=== Short-Term Memory

Conversation messages are stored as linked nodes in Neo4j. Each message gets a vector embedding, so the agent can do semantic search across past conversations, not just retrieve the last N messages. If a user mentioned "Blade Runner" three sessions ago, short-term memory can surface that conversation when the topic comes up again.

=== Long-Term Memory

Entities, preferences, and facts are extracted from conversations using the POLE+O data model (Person, Object, Location, Event, Organization). When a user says "I love Christopher Nolan films," long-term memory stores that as a preference. When the agent discusses "Inception," it stores the movie as an entity linked to its director, genre, and the conversation where it was mentioned.

This extraction happens automatically. Over time, the agent builds a graph of knowledge about each user and the topics they discuss.

=== The ReAct Pattern: Reason + Act

Agents don't execute a single action and stop. They loop through a cycle known as ReAct (Reason + Act):

* **Reason** -- observe the current state (user input, tool results, conversation history) and decide what to do next
* **Act** -- execute an action: call a tool, query an API, or generate a response
* **Observe** -- examine the result and feed it back into the next reasoning step

The agent repeats this loop until the task is complete. Each pass produces a trace — what the agent reasoned, what it did, and what came back.

=== Reasoning Memory

Reasoning memory captures those traces. Each trace records the task, the steps taken, the tools called, and the outcome. When the agent encounters a similar task later, it can retrieve those traces and reuse strategies that succeeded or avoid approaches that failed.

== How Neo4j Stores It

All three memory types live in the same Neo4j database as graph structures. Messages form chains linked to sessions. Entities become nodes with typed relationships (PERSON, ORGANIZATION, OBJECT). Reasoning traces connect to the steps and tool calls that compose them.

Because everything is in the same graph, memories connect to each other. An entity extracted from a conversation links back to the message it came from, the session it occurred in, and other entities it relates to. This is more than storage; it's a connected knowledge structure the agent can traverse.

== The neo4j-agent-memory Package

The MAF context provider you will use in this module is built on the `neo4j-agent-memory` library. The library handles all the underlying memory operations (storing conversations, extracting entities, managing reasoning traces) while the Microsoft Agent Framework integration layer wraps those capabilities into the `BaseContextProvider` lifecycle you learned about in Module 2.

The integration provides four components:

* `MemoryClient` -- connects to Neo4j and exposes all three memory types
* `Neo4jMicrosoftMemory` -- wraps the client into a unified interface for MAF agents
* `Neo4jContextProvider` (memory variant) -- implements `before_run()`/`after_run()` for automatic memory injection and extraction
* `create_memory_tools()` -- generates callable tools for explicit memory operations

The next lesson uses these components to build a memory context provider. Subsequent lessons cover entity extraction, active memory tools, reasoning traces, and graph algorithm integration.
