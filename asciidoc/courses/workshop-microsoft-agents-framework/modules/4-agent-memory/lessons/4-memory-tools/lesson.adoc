:type: lesson

== Passive Memory

The memory context provider from link:../2-memory-context-provider/[Lesson 4-2] handles memory passively. It injects relevant memories before each turn and stores new ones after. The agent never decides to look something up or save a preference; the framework does it automatically.

That works well for general context, but sometimes the agent needs to take a deliberate action. A user says "remember that I hate horror movies" and expects the agent to explicitly store that preference. Or the agent needs to search for a specific entity in the knowledge graph before answering. Passive injection alone can't handle these cases because the agent has no way to trigger a targeted memory operation.

== Agent Memory Also Provides Tools

In addition to the context provider, the agent memory module provides **tools**: functions the agent can call on demand to actively manage its memory.

The context provider runs automatically every turn. Tools run only when the agent decides to use them.

This means the agent gets two complementary mechanisms:

* **Context provider**: background memory that just works
* **Memory tools**: targeted operations the agent controls

== What Are Tools?

In the Microsoft Agent Framework, a **tool** is a callable function exposed to the LLM. When the agent decides it needs to perform an action such as searching a database, saving a preference, or looking up an entity, it calls a tool.

Tools are defined as `FunctionTool` instances and passed to the agent's `tools` list. The LLM sees the tool's name, description, and parameters, and can invoke it during a conversation turn.

Memory tools work the same way as any other agent tool. The only difference is they operate on the agent's memory store.

== The Six Memory Tools

`create_memory_tools()` generates callable `FunctionTool` instances bound to your memory instance:

[source,python]
----
async def main():
    memory_client = MemoryClient(settings)
    await memory_client.connect()

    try:
        # Create unified memory interface
        memory = Neo4jMicrosoftMemory.from_memory_client(
            memory_client=memory_client,
            session_id="movie-tools-session",
            include_short_term=True,
            include_long_term=True,
            extract_entities=True,
        )

        # Create callable memory tools
        tools = create_memory_tools(memory)

        for tool in tools:
            print(f"  - {tool.name}: {tool.description[:60]}...")


        client = OpenAIResponsesClient()

        agent = client.as_agent(
            name="movie-assistant",
            instructions=(
                "You are a movie recommendation assistant with persistent memory.\n\n"
                "IMPORTANT: You MUST call the remember_preference tool every time "
                "a user states a preference, favorite, or interest. Do NOT just "
                "acknowledge it verbally. You must actually invoke the tool to "
                "persist it. For example, if a user says they love sci-fi, call "
                "remember_preference with category='genre' and preference='loves sci-fi'.\n\n"
                "When making recommendations, use recall_preferences to check "
                "what the user likes before suggesting something.\n\n"
                "You have access to a knowledge graph of movies and "
                "your memory of past conversations."
            ),
            tools=tools,
            context_providers=[memory.context_provider],
        )

        session = agent.create_session()


        queries = [
            "I love Christopher Nolan movies and anything about space.",
            "What are my movie preferences?",
            "Based on what you know about me, what should I watch next?",
        ]

        for query in queries:
            print(f"\nUser: {query}")
            async for update in agent.run(query, stream=True, session=session):
                if update.text:
                    print(update.text, end="", flush=True)
            print()


        results = await memory.search_memory(
            query="Christopher Nolan movies space",
            include_messages=True,
            include_entities=True,
            include_preferences=True,
            limit=5,
        )

        print("=== Stored Memories ===\n")

        if results.get("preferences"):
            print("Preferences:")
            for pref in results["preferences"]:
                print(f"  [{pref['category']}] {pref['preference']}")
            print()

        if results.get("entities"):
            print("Entities:")
            for entity in results["entities"][:5]:
                print(f"  {entity['name']} ({entity['type']})")
            print()

        if results.get("messages"):
            print(f"Messages stored: {len(results['messages'])}")
    finally:
        await memory_client.close()

asyncio.run(main())

----

This produces six tools:

* `search_memory` -- searches across all memory types for a given query
* `remember_preference` -- saves a user preference (e.g., category "genre", preference "prefers sci-fi")
* `recall_preferences` -- retrieves saved preferences matching a topic
* `search_knowledge` -- searches the entity graph for people, movies, genres, and other extracted entities
* `remember_fact` -- stores a factual statement as a subject-predicate-object triple
* `find_similar_tasks` -- retrieves past reasoning traces for similar tasks

The agent uses these the same way it uses any tool: by examining the function name and description, then deciding whether to call one based on the user's query.

== Context Provider vs Memory Tools

The two mechanisms serve different roles:

[cols="1,2,2", options="header"]
|===
| Aspect | Context Provider | Memory Tools

| **Invocation**
| Automatic every turn
| Agent decides when

| **Direction**
| Recall and store
| Search, save, recall on demand

| **User visibility**
| Transparent
| Visible in agent reasoning

| **Control**
| Framework-managed
| Agent-managed
|===

The context provider says "I automatically remember relevant things." Memory tools say "I can explicitly search, save, and recall."

== The Combined Pattern

The most effective setup uses both mechanisms together. Pass the context provider and the tools to the agent. Notice the explicit, prescriptive instructions. Without these, the LLM may acknowledge a preference verbally but skip calling the `remember_preference` tool, so the preference is never persisted:

[source,python]
----
client = OpenAIResponsesClient()

agent = client.as_agent(
    name="movie-assistant",
    instructions=(
        "You are a movie recommendation assistant with persistent memory.\n\n"
        "IMPORTANT: You MUST call the remember_preference tool every time "
        "a user states a preference, favorite, or interest. Do NOT just "
        "acknowledge it verbally. You must actually invoke the tool to "
        "persist it. For example, if a user says they love sci-fi, call "
        "remember_preference with category='genre' and preference='loves sci-fi'.\n\n"
        "When making recommendations, use recall_preferences to check "
        "what the user likes before suggesting something.\n\n"
        "You have access to a knowledge graph of movies and "
        "your memory of past conversations."
    ),
    tools=tools,
    context_providers=[memory.context_provider],
)

session = agent.create_session()
----

The agent now operates on two levels simultaneously. Background memory arrives automatically through the context provider. Targeted memory operations happen through explicit tool calls when the agent decides they are needed.

== Run a Conversation

[source,python]
----
queries = [
    "I love Christopher Nolan movies and anything about space.",
    "What are my movie preferences?",
    "Based on what you know about me, what should I watch next?",
]

for query in queries:
    print(f"\nUser: {query}")
    async for update in agent.run(query, stream=True, session=session):
        if update.text:
            print(update.text, end="", flush=True)
    print()
----

Here is what happens on each turn:

1. **Turn 1** -- The user expresses preferences. The agent recognizes this and calls `remember_preference` twice, once for "Christopher Nolan movies" and once for "space themes." Separately, the context provider's `after_run()` saves the message and extracts "Christopher Nolan" as a PERSON entity.
2. **Turn 2** -- The user asks about their preferences. The agent calls `recall_preferences` to look up saved preferences and reports them. It could also get some of this from the passive context injected by `before_run()`, but the explicit tool call gives it a targeted, complete answer.
3. **Turn 3** -- Both mechanisms contribute. `before_run()` injects relevant memories as background context. The agent can also call `search_knowledge` to find movies linked to Christopher Nolan in the entity graph, then `recall_preferences` to confirm the user likes space themes, before generating a recommendation.

== Verify Stored Memories

After the conversation, you can inspect what the memory system has stored. Use `search_memory()` to query across all memory types:

[source,python]

This confirms that the combined context provider and tools approach has persisted preferences, extracted entities, and stored conversation messages in Neo4j.

== Experiment

Test the interaction between passive and active memory:

* Tell the agent you dislike a genre, then ask for recommendations. Does it avoid that genre?
* Ask the agent to remember a specific fact ("The Matrix was released in 1999") and verify it later with a question.
* End the session, create a new one with the same `session_id`, and check whether long-term memories (preferences, entities) persist. Short-term messages from the old session should also be searchable.
* Ask the agent a question it struggled with before. If reasoning traces are recorded, it may approach the task differently.

== Lesson Summary

The memory context provider handles passive memory automatically. In addition, agent memory provides tools for targeted operations. Tools are callable functions the agent invokes on demand. The combined pattern uses both: passive context for the common case, active tools for specific operations. Both mechanisms persist data in the same Neo4j graph.
