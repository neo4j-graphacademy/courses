= Neo4j Knowledge Retrieval Context Provider Overview
:order: 1
:type: lesson

The context provider you built from scratch in Module 2 managed its own state and injected instructions based on simple logic. Production knowledge graph retrieval needs more than that, including index-based search, embeddings, configurable result formatting, and support for different search strategies.

The link:https://github.com/neo4j-labs/neo4j-maf-provider[`agent-framework-neo4j`^] package provides `Neo4jContextProvider`, a ready-made context provider that connects to a Neo4j database, searches an index, and injects the results before every LLM call. It extends the same `BaseContextProvider` you used earlier but handles the search, embedding, and formatting internally.

All retrieval uses the link:https://github.com/neo4j/neo4j-graphrag-python[`neo4j-graphrag-python`^] library, Neo4j's official Python package for building retrieval-augmented generation (RAG) pipelines. The `neo4j-graphrag` package provides retriever classes such as `VectorRetriever`, `HybridRetriever`, `VectorCypherRetriever`, and others that know how to query Neo4j vector and fulltext indexes and optionally enrich results with graph traversals. `Neo4jContextProvider` selects the correct retriever based on your `index_type` configuration, wraps it in the MAF lifecycle, and handles the async execution. You configure the provider; `neo4j-graphrag` runs the search.

Beyond retrieval, `neo4j-graphrag-python` also provides a complete pipeline for _building_ knowledge graphs from unstructured text. Its `SimpleKGPipeline` automates the end-to-end process: splitting documents into chunks, using an LLM to extract entities and relationships guided by a schema, computing vector embeddings for semantic search, and storing everything in Neo4j. It even handles entity resolution to merge duplicates across the graph. This means the same library you use to query your knowledge graph can also construct it, taking raw PDFs, articles, or other text and turning them into the structured graph data that your retrievers search against.

For full details on the retriever classes, knowledge graph construction pipelines, and other capabilities, see the link:https://neo4j.com/docs/neo4j-graphrag-python/current/[neo4j-graphrag-python documentation^].

== How It Works

`Neo4jContextProvider` implements `before_run()` with the following steps:

1. Filters incoming messages to user and assistant roles
2. Takes the most recent messages (configurable with `message_history_count`)
3. Concatenates the message text into a search query
4. Executes a search against a Neo4j index based on the configured `index_type`
5. Formats the results and injects them as context messages using `context.extend_messages()`

The provider doesn't extract entities or keywords from the query. It passes the full message text to the search index and lets Neo4j handle relevance ranking.

== Three Search Modes

The `index_type` parameter determines which search strategy the provider uses:

[cols="1,2,2"]
|===
| Mode | How It Works | Key Strength

| `"vector"`
| Embeds the query, searches a vector index by cosine similarity
| Understands meaning

| `"fulltext"`
| Tokenizes the query, removes stop words, searches a fulltext index by BM25 ranking
| Matches keywords

| `"hybrid"`
| Runs both vector and fulltext search, combines the scores
| Combines both
|===

Each mode maps to a retriever class from the link:https://neo4j.com/docs/neo4j-graphrag-python/current/[`neo4j-graphrag-python`^] library:

* `"vector"` uses link:https://neo4j.com/docs/neo4j-graphrag-python/current/api.html#vectorretriever[`VectorRetriever`^] (or `VectorCypherRetriever` when a `retrieval_query` is provided)
* `"fulltext"` uses a fulltext retriever that queries Neo4j's Lucene-based fulltext indexes
* `"hybrid"` uses link:https://neo4j.com/docs/neo4j-graphrag-python/current/api.html#hybridretriever[`HybridRetriever`^] (or `HybridCypherRetriever` with a `retrieval_query`)

You configure the mode when creating the provider. `Neo4jContextProvider` creates the appropriate `neo4j-graphrag` retriever on connection and delegates every search call to it.

Each provider instance supports a single `index_type`. If you need multiple search strategies, for example vector search on one index and fulltext search on another, create separate provider instances and pass them all to the agent's `context_providers` list. The agent merges the context from every provider before calling the LLM.

== Graph-Enhanced Search

Any search mode can be graph-enhanced by adding a `retrieval_query`. This layers graph traversal on top of the initial index search:

* **Initial search** performs vector, fulltext, or hybrid search as configured
* **Graph traversal** executes a Cypher query from each result to follow connections
* **Richer context** returns related entities, relationships, and metadata alongside the matched text

Without graph enrichment, you get isolated text chunks. With graph enrichment, you get text chunks plus structured knowledge from the graph.

Graph enrichment is what separates GraphRAG from standard vector RAG. The dedicated graph-enriched retrieval lesson covers this in detail.

== Key Parameters

[cols="1,3",options="header"]
|===
| Parameter | Description

| `uri`, `username`, `password`
| Neo4j connection details. Falls back to `NEO4J_URI`, `NEO4J_USERNAME`, `NEO4J_PASSWORD` environment variables if not provided.

| `index_name`
| The name of the Neo4j index to query. Required.

| `index_type`
| One of `"vector"`, `"fulltext"`, or `"hybrid"`. Defaults to `"vector"`.

| `top_k`
| Number of results to retrieve. Defaults to 5.

| `context_prompt`
| Text prepended to search results to guide the LLM.

| `message_history_count`
| Number of recent messages to include in the search query. Defaults to 10.

| `embedder`
| Required for `"vector"` and `"hybrid"` modes. An object implementing the `neo4j-graphrag` `Embedder` interface.

| `fulltext_index_name`
| Required for `"hybrid"` mode. The name of the fulltext index to combine with vector search.

| `filter_stop_words`
| For `"fulltext"` mode. Removes common words (articles, prepositions, question words) before searching. Defaults to `True`.

| `retrieval_query`
| Optional Cypher query for graph enrichment after index search. Works with all three modes.
|===
