[.question]
= When a Cypher Template Is Not Enough

For "Which customers ordered products from more than two different suppliers?", why can't you implement this with a single Cypher Template that uses only `$parameter` placeholders for variable values?

* [ ] Because the question mentions "customers" and "suppliers," so it needs semantic search
* [x] Because the query structure itself depends on the question: the filter count, the multi-hop path, and the aggregation cannot be fixed in advance
* [ ] Because aggregations are not supported in Cypher Templates
* [ ] Because the graph might not contain the required relationship types

[TIP,role=hint]
.Hint
====
Apply the lesson test: can you write the complete Cypher right now with only `$parameter` slots? What varies between questions of this type—just values, or the MATCH pattern and WHERE conditions too?
====

[TIP,role=solution]
.Solution
====
**The query structure depends on the question.**

For this question, the filter ("more than two"), the multi-hop path (Customer → Order → Product → Supplier), and what you aggregate and return all depend on the specific question. You cannot pre-write one query and plug in parameters. A Cypher Template assumes a fixed structure with only values changing. Here the structure itself is dynamic, so the agent needs a tool that generates Cypher at runtime.

**Why the others are wrong:**

* Semantic search: Similarity Search finds nodes by vector similarity; it does not generate Cypher for multi-hop aggregations.
* Aggregations in templates: Cypher Templates support aggregations when the pattern is fixed (e.g. "top N by count" with a single `$n` parameter).
* Relationship types: The issue is whether the query shape can be fixed in advance, not whether the schema supports the relationships.

**Recall:** Use Text2Cypher when the MATCH pattern, WHERE conditions, or RETURN clause change between questions; use Cypher Template when only parameter values change.
====
