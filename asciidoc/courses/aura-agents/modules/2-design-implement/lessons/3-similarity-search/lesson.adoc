= Similarity Search
:order: 3
:type: lesson

Similarity Search finds graph nodes that are semantically similar to the user's question using vector embeddings. It converts the user's input into a vector embedding, then compares it against embeddings stored as node properties to surface semantically related results.

Similarity Search works well when you can't describe what you're looking for with exact words. Good examples are "products like spicy condiments" or "find items similar to this description."

Your graph must have vector embeddings on the nodes you want to search, and a vector index must exist. Supported embedding providers include OpenAI (through Azure) and Vertex AI.

This course focuses on Cypher Template and Text2Cypher. Similarity Search is available when your graph has embeddings, but no hands-on configuration is included for it here.

== Choosing Between Tools

Apply this test to every question type: can you write the complete Cypher query right now, with only `$parameter` slots for variable values?

[cols="1,1,2"]
|===
|Answer |Tool |Reasoning

|Yes: MATCH, WHERE, and RETURN are fixed. Only values change.
|**Cypher Template**
|"Get customer ALFKI" → `MATCH (c:Customer {id: $id})`. The pattern is fixed, only the ID changes. Write it as a template.

|No: the query structure itself changes
|**Text2Cypher**
|"Which customers ordered products from more than two different suppliers?": the filter count, the multi-hop pattern, and what to return all depend on the question. You cannot write this in advance.

|You need semantic matching, not graph traversal
|**Similarity Search**
|"Products similar to spicy condiments": no exact property value to match on. Requires vector similarity. Does not traverse relationships.
|===

Aggregations are a common source of confusion. "Top 10 customers by order count" sounds complex, but the query structure is completely fixed. It takes one `$limit` parameter and nothing else changes. Write it as a Cypher Template. Text2Cypher is for questions where the `MATCH` pattern or `WHERE` clause itself depends on what the user asks.

For relationship traversal, a Cypher Template is a better choice than Similarity Search. A template follows `PLACED` to orders and then `CONTAINS` to products in a single query. Similarity Search, by contrast, finds nodes that are semantically close to the input. It answers "what is like this?" not "what is connected to this?"
