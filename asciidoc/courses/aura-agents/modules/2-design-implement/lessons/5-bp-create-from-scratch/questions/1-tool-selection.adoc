= Why This Question Fits a Template

"Show me the top 10 customers by order count" is a good fit for a Cypher Template. What is the main reason?

* [ ] Aggregation queries always use Cypher Templates so that the LLM can optimize the query plan
* [x] The full query structure is fixed—same MATCH, COUNT, ORDER BY, and LIMIT pattern—and only the numeric limit (e.g. 10) varies, so you can use a single template with a `$n` parameter
* [ ] Similarity Search cannot rank by count, so a template is the only option
* [ ] Text2Cypher is reserved for questions that mention more than two entity types

[TIP,role=hint]
.Hint
====
Apply the test from the lesson: can you write the complete Cypher query right now, with only `$parameter` slots for variable values? What stays the same every time someone asks for "top N by order count"?
====

[TIP,role=solution]
.Solution
====
**The query structure is fixed; only the limit value changes.**

You can write one query: match customers, count orders, order by count descending, limit `$n`. The MATCH pattern, aggregation, and return shape are identical for "top 5," "top 10," or "top 20." That is exactly when a Cypher Template is appropriate—when the full structure is predictable and only specific values (here, the limit) need to be filled from the user's question.

**Why the others are wrong:**

* Query plan: The choice between Cypher Template and Text2Cypher is about whether the query shape can be pre-defined, not about Neo4j's query planning.
* Similarity Search: True that Similarity Search does not rank by count, but the key reason for using a template here is the fixed query structure, not the absence of another tool.
* Entity types: Text2Cypher is for questions where the MATCH pattern or filters change between questions, not determined by how many entity types are mentioned.

**Recall:** The question types table in this lesson defines the test: if the full query structure is fixed, use a Cypher Template.
