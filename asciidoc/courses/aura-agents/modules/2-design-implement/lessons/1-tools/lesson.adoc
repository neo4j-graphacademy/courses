= Available Tools
:order: 1
:type: lesson
== Introduction

The tools you configure on an Aura Agent determine what data it can retrieve and how. Choosing the correct tool for each question type is the primary design decision when building an agent.

The knowledge graph your agent queries contains two kinds of information:

* **Structured, connected data**: nodes and relationships that tools query with Cypher
* **Facts and domain state**: the specific data about your business or domain that makes answers accurate and relevant

In this lesson, you will learn:

* What each tool does and when to use it
* How to choose between tools for a given question type


== Cypher Template

A **Cypher Template** is a pre-written, parameterized Cypher query. You define the query once and the LLM extracts parameter values from the user's question and runs it.
Queries use named `$parameter` placeholders; the LLM fills them from the user's question at runtime.

Use a Cypher Template for common, repeated questions with known structure: "Get customer ALFKI", "List products in Beverages", "Top 10 customers by order count." Because the query is pre-written, results are deterministic: the same question always runs the same Cypher.

[TIP]
.Build templates first
====
Start with Cypher Templates for your core question types. Each template you add reduces how often the agent needs to generate Cypher dynamically.
====


=== What is a parameter?

A **parameter** is a named placeholder in a Cypher query, prefixed with `$`, that gets its value supplied at runtime.
Instead of hardcoding a value in the query, you write `$parameter_name` and the LLM extracts the correct value from the user's question and passes it in.

For example, this query has one parameter, `$customer_id`:

[source,cypher,role=noplay]
----
MATCH (c:Customer {id: $customer_id})
RETURN c.companyName, c.contactName, c.city
----

When a user asks "Get me customer ALFKI", the LLM extracts `"ALFKI"` from the question and runs the query with `$customer_id = "ALFKI"`.
The same template handles "Get me customer QUICK" by substituting `"QUICK"`, with no new query required.

Parameters can hold different data types:

[cols="1,2,2"]
|===
|Type |Example parameter |Example question

|**string**
|`$customer_id`
|"Get customer ALFKI" → `customer_id = "ALFKI"`

|**string**
|`$category`
|"List products in Beverages" → `category = "Beverages"`

|**int**
|`$limit`
|"Show me the top 5 customers" → `limit = 5`
|===

Each parameter needs a name, a data type, and a description.
The LLM uses the description to know what value to extract. Treat it as an instruction, not a label.
A description like `"The customer ID"` is weaker than `"The Northwind customer ID, for example ALFKI or QUICK"`.


== Text2Cypher

Text2Cypher converts a natural language question into a Cypher query at runtime and executes it against your graph.

Use Text2Cypher when the query structure itself changes between questions: different `MATCH` patterns, dynamic filter combinations, or aggregations over properties you cannot define in advance.

A useful test: can you write the complete Cypher right now, with only `$parameter` slots for variable values? If the `MATCH` pattern, `WHERE` conditions, and `RETURN` clause are all fixed and only the values change, that is a Cypher Template. If the shape of the query depends on what the user asks, use Text2Cypher.

Text2Cypher uses a fine-tuned LLM to generate Cypher at runtime, so the output is probabilistic. The same question can produce different Cypher on different calls, and generated queries may contain errors.

[NOTE]
.Text2Cypher in production
====
Text2Cypher relies on a language model and can produce queries with errors. Test generated queries in the reasoning panel before relying on them for production use.
====


== Similarity Search

Similarity Search finds graph nodes that are semantically similar to the user's question using vector embeddings. It converts the user's input into a vector embedding, then compares it against embeddings stored as node properties to surface semantically related results.

Similarity Search works well when you can't describe what you're looking for with exact words. Good examples are "products like spicy condiments" or "find items similar to this description."

Your graph must have vector embeddings on the nodes you want to search, and a vector index must exist. Supported embedding providers include OpenAI (through Azure) and Vertex AI.

This course focuses on Cypher Template and Text2Cypher. Similarity Search is available when your graph has embeddings, but no hands-on configuration is included for it here.


== Choosing Between Tools

Apply this test to every question type: can you write the complete Cypher query right now, with only `$parameter` slots for variable values?

[cols="1,1,2"]
|===
|Answer |Tool |Reasoning

|Yes: MATCH, WHERE, and RETURN are fixed. Only values change.
|**Cypher Template**
|"Get customer ALFKI" → `MATCH (c:Customer {id: $id})`. The pattern is fixed, only the ID changes. Write it as a template.

|No: the query structure itself changes
|**Text2Cypher**
|"Which customers ordered products from more than two different suppliers?": the filter count, the multi-hop pattern, and what to return all depend on the question. You cannot write this in advance.

|You need semantic matching, not graph traversal
|**Similarity Search**
|"Products similar to spicy condiments": no exact property value to match on. Requires vector similarity. Does not traverse relationships.
|===

Aggregations are a common source of confusion. "Top 10 customers by order count" sounds complex, but the query structure is completely fixed. It takes one `$limit` parameter and nothing else changes. Write it as a Cypher Template. Text2Cypher is for questions where the `MATCH` pattern or `WHERE` clause itself depends on what the user asks.

For relationship traversal, a Cypher Template is a better choice than Similarity Search. A template follows `PLACED` to orders and then `CONTAINS` to products in a single query. Similarity Search, by contrast, finds nodes that are semantically close to the input. It answers "what is like this?" not "what is connected to this?"
