== Introduction

Aura Agents let you build agents that query your knowledge graph from the Aura console without writing code. Understanding what they are and when to use them helps you choose the correct approach for your application.

In this lesson, you will learn:

* What Aura Agents are and their components
* The agent lifecycle
* When to use Aura Agents

== What are Aura Agents?

A **Neo4j Aura Agent** is a no/low-code agent platform built into the Aura Console. The orchestration, reasoning, and graph queries all run inside Aura's infrastructure with no external API integration or application code required. You connect the agent to an AuraDB instance, configure tools in the console, and the agent handles the rest.

Conversation history is not stored between sessions.

== When to use Aura Agents

Aura Agents are useful when:

* You want to expose knowledge graph data using natural language, without writing application code
* You need to integrate graph-backed answers into Claude Desktop or Cursor through MCP

Aura Agent tools query only your knowledge graph at an instance level. To combine your knowledge graph data with external APIs, build a separate AI application or a multi-agent system.


// === Agent processing and planning

// Every Aura Agent runs on **Gemini 2.5 Flash**, which powers the core reasoning loop behind every request.

// Gemini 2.5 Flash handles four responsibilities:

// * **Orchestration**: coordinates the sequence of steps from user input to final answer, deciding when to call a tool and when to respond
// * **Reasoning**: interprets user intent, maps it to the available tools, and determines what information is needed to answer the question
// * **Tool selection**: chooses the correct tool (Cypher Template, Text2Cypher, or Similarity Search) for each question, based on the tool descriptions you write
// * **Chain-of-thought transparency**: exposes its reasoning steps in the agent's reasoning panel, so you can inspect which tools were considered, which were selected, and why

// The model is managed and updated by Neo4j. See link:https://neo4j.com/docs/reference/license/agent-ai-model/[How Aura Agent uses AI models^] for the full model reference.


== Aura Agent architecture

Aura Agents have four components:

* **LLM** for planning, reasoning, and tool selection
* **Tools** for querying the structured data in AuraDB (read-only)
* **Knowledge graph**, the nodes and relationships your tools traverse
* **Orchestrator** to coordinate the workflow

An Aura Agent's role is strictly **retrieval**: it selects the correct tool, queries the structured data in your graph's nodes and relationships, and returns an answer grounded in data you control, not in what the LLM was trained on.

== The Aura Agent lifecycle

An Aura Agent does not answer in one step. It follows five steps:

. **Receive**: User query or message
. **Reason**: LLM plans which tools to use and in what order
. **Act**: Invoke one of the available tools: a Cypher Template runs a pre-written Cypher query, Text2Cypher generates Cypher from the user's question at runtime, or Similarity Search finds nodes by vector similarity
. **Observe**: Incorporate tool results into context
. **Respond**: Generate final answer or take next action

The diagram below illustrates this workflow: from user input through reasoning, tool selection and execution, to final output.

image::images/agent-process.svg[Agent workflow: user input, reasoning, tool selection, tool execution, and final output]

At each step, the agent is building context: it identifies what the user needs, picks the tool most likely to answer it, runs the query, and incorporates the results into its context before generating a response.
