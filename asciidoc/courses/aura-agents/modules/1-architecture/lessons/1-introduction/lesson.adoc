== Introduction

Aura Agents let you build GraphRAG-style agents in the Aura console without writing code. Understanding what they are and when to use them helps you choose the right approach for your application.

In this lesson, you will learn what Aura Agents are, when to use them, and how they fit into the Neo4j ecosystem.

[.slide]
== What are Aura Agents?

A **Neo4j Aura Agent** is an independent software entity that understands user intent and invokes the right tools to achieve a goal.

The agent interprets your question, finds relevant information in the graph, and invokes the appropriate tool: Similarity Search, Text2Cypher, or Cypher Template.

An Aura Agent consists of:

* **LLM** for reasoning and response generation
* **Tools** for querying the graph: Cypher Template, Similarity Search, and Text2Cypher (read-only)
* **Knowledge base** (your Neo4j graph)
* **Orchestrator** to coordinate the workflow

Conversation history is not currently stored; each turn is stateless from the platform's perspective. For the full reference, see link:https://neo4j.com/docs/aura/aura-agent/[Aura Agent documentation^].


[.slide]
== Prerequisites and Pricing

To create an Aura Agent:

* Have a knowledge graph in a **running AuraDB instance**
* Enable **Generative AI assistance** in your Organization settings
* Toggle on **Aura Agent** (available when Generative AI assistance is enabled)

[WARNING]
.Keep Generative AI assistance enabled
====
Disabling Generative AI assistance automatically disables Aura Agent.
====

**Pricing**: Internal agents are free. External agents, exposed via REST API or MCP server, incur charges. See link:https://neo4j.com/pricing/[Neo4j pricing^].

**Location**: All Aura Agents run in Belgium (GCP `europe-west1`). Agent interactions route through this region regardless of access mode.

**AI models**: Aura Agent uses a curated set of AI models for planning, reasoning, retrieval, and explainability. See link:https://neo4j.com/docs/reference/license/agent-ai-model/[How Aura Agent uses AI models^].

**Construction** is where you draft agents with domain-specific tools and prompts. Agents often struggle here when schemas or tool descriptions are unclear. **Grounding Failure** happens when agents misinterpret data or pick the wrong tools. **Runtime** covers deployment, testing, and iteration. Aura Agents address all three: you get structured tools, the schema is introspectable, and the runtime is fully managed.

== Agentic Concepts

* **Agents**: Systems that perceive, reason, and act autonomously
* **Tools**: Capabilities the agent invokes: Cypher Template, Similarity Search, and Text2Cypher
* **Retrieval**: GraphRAG and vector search to fetch relevant context

Agents are useful when:

* Queries require multi-hop reasoning across relationships
* You need to combine graph data with other data sources (e.g., via your application layer)
* Users ask natural language questions over complex, connected data

== When to Use Agents

Use an agent when your application needs:

* **Multi-hop reasoning**: "Which customers bought products from suppliers in the same region as their top-selling category?"
* **Graph plus application logic**: Your application can use the agent for graph queries and combine results with other data or APIs in your own code
* **Natural language queries**: "Summarize sales patterns for customers in Germany"
