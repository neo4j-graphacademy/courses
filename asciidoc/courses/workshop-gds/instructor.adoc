== General note 
The length of the workshop is 4 hours -- in theory. That assumes that participants are relatively familiar with many of the concepts and technical approaches already. 

That said, the workshop is designed to frontload core details early, and then focus of practical applications and real-world use cases. 

If users struggle to keep up, or ask a lot of questions, don't feel obliged to cover the entire workshop in the allotted time. If a user is struggling, feel free to slow down. 

It's better to give a novice user a thorough understanding of the core concepts and practical applications, than to rush through the content and leave them with a sense of confusion and frustration.

== Suggested Pacing

*Module 1* is the densest -- 8 lessons covering foundational concepts. Lessons 4 (Graph Projections and Structure) and 5 (Projection Practice) are the most important and deserve the most time. Lessons 7-8 (Execution Modes, Projection Configuration) can be covered more briskly if time is tight.

*Module 2* builds to a satisfying payoff: 211 previously unknown fraud risk users identified, representing ~13% of all P2P transaction volume. Let participants appreciate each step building toward that result.

*Module 3* is the longest module. The first 4 lessons teach Python GDS mechanics; lessons 5-9 apply them to a citation network. If short on time, you can demo lessons 8-9 (Louvain and FastRP on citations) rather than having everyone run them.

*Module 4* is the lightest. The key takeaway is the Aura Graph Analytics workflow (ephemeral sessions, remote projection) -- the pathfinding algorithms themselves are straightforward.

== Common Misconceptions

* *"GDS runs on the stored graph directly"* -- No, GDS operates on an in-memory projection. This is a critical concept to reinforce throughout Module 1.
* *"More nodes in the projection = better results"* -- Projecting everything usually produces noise. Targeted projections are key.
* *"Louvain gives deterministic results"* -- It doesn't. This is why Module 2 introduces WCC as a deterministic alternative.
* *"PageRank and Betweenness measure the same thing"* -- PageRank measures influence (being cited by important papers); Betweenness measures bridging (connecting different areas). Module 3 contrasts these directly.
* *"Undirected = no direction matters"* -- Some algorithms require undirected projections. This trips people up when projections silently produce wrong results.

== Key Moments to Emphasise

* *Module 1, Lesson 2:* The Project -> Run -> Write workflow. Everything else builds on this.
* *Module 2, Lesson 3:* The 98% search space reduction using Louvain. This is the "aha" moment for the fraud use case.
* *Module 2, Lesson 5:* Encoding domain hypotheses as graph relationships -- this is the creative step that separates a demo from real-world application.
* *Module 3, Lesson 7:* The PageRank vs Betweenness scatter plot. Great visual for showing different definitions of "important."
* *Module 4, Lesson 2:* Comparing historical routes against Dijkstra's optimal path. Concrete business value.

== FAQ from Participants

*"Do I need GDS installed locally?"* -- No, the workshop uses sandboxed environments and Codespaces.

*"Is the Python client the only way to use GDS programmatically?"* -- No, you can also use Cypher directly. The Python client wraps it with convenience methods and DataFrame integration.

*"What's the difference between GDS and Aura Graph Analytics?"* -- GDS is the library; Aura Graph Analytics is a cloud service that runs GDS on ephemeral compute sessions, keeping your production database unaffected.

*"Can I use these algorithms on my own data?"* -- Yes. The workflow is identical: project a subgraph, run algorithms, write results back.
