## Step 3 — Choose the most relevant graph pattern for their model

Look at the user's node labels and relationship types, then pick whichever of these patterns fits best:

### Pattern A — Collaborative filtering
*When to use:* The model has a customer/user node connected to items (products, movies, songs) via a purchase, view, or rating relationship.
*Business question:* What did people with similar behaviour choose that this person hasn't seen yet?
*Shape:* `(a)-[:ACTION]->(item)<-[:ACTION]-(b)-[:ACTION]->(unseen)` where `(a)` has not already seen `unseen`.

### Pattern B — Friend-of-friend / people you may know
*When to use:* The model has a self-referential relationship on a person/user node (FOLLOWS, FRIENDS_WITH, CONNECTED_TO).
*Business question:* Who is two hops away but not yet directly connected?
*Shape:* `(a)-[:REL]->(mid)-[:REL]->(fof)` where `(a)` is not already connected to `fof`.

### Pattern C — Degree centrality / most connected
*When to use:* Any model — find the most-connected or most-active nodes.
*Business question:* Which nodes have the highest number of direct relationships?
*Shape:* `MATCH (n:Label) RETURN n.property, count { (n)-[:REL]-() } AS degree ORDER BY degree DESC`

### Pattern D — Co-occurrence / shared context
*When to use:* The model has two nodes of the same type that both connect to a shared intermediate node (actors in films, authors on papers, accounts at merchants).
*Business question:* Which pairs appear together most often?
*Shape:* `(a)-[:REL]->(shared)<-[:REL]-(b)` grouped by the pair `(a, b)`.

### Pattern E — Variable-length path / reachability
*When to use:* The model has a chain of relationships suitable for spreading influence, risk, or connection (fraud rings, org hierarchies, recommendation chains).
*Business question:* What nodes are reachable from a starting node within N hops?
*Shape:* `(start)-[:REL*1..N]->(reachable)`

## Example of correct adaptation

If the user's model has `Customer`, `Product`, and `PURCHASED`, a collaborative filtering query looks like:

```cypher
// "Which products do customers with similar purchase history buy that this customer hasn't tried?"
MATCH (c:Customer {id: $customerId})-[:PURCHASED]->(p:Product)<-[:PURCHASED]-(similar:Customer)
MATCH (similar)-[:PURCHASED]->(rec:Product)
WHERE NOT (c)-[:PURCHASED]->(rec)
RETURN rec.name AS recommendation, count(similar) AS score
ORDER BY score DESC
LIMIT 10
```

If the user's model has `User` and `FOLLOWS` instead, adapt accordingly — never mix labels from the example with the user's actual model.
