[.slide.discrete]
== Introduction

In Microsoft Agent Framework, an **agent** is a program object that combines a **language model**, **instructions**, and **tools**. A **tool** is anything the model can invoke with structured arguments: HTTP APIs, database clients, search, **MCP servers**, or your own functions.

**Orchestration** is how you connect multiple steps: sequential chains, branches, parallel fan-out, or explicit hand-offs between agents. The framework expresses those patterns in code and, in many samples, as **graph-shaped** workflows.

[.slide]
== Agents and tools

**Agent:** Holds policy text (“what you are allowed to do”), model configuration, and a **tool registry**. It decides *when* to call tools based on user input and prior turns.

**Tool:** A narrow, testable surface. Good tools return **typed** or **JSON-friendly** results so the model can reason over them. Heavy business logic still belongs in **your** libraries; the tool is the adapter.

This separation mirrors what you already did in the GenAI workshop: retrievers and Cypher-backed helpers behave like **tools** feeding the conversation.

[.slide]
== Model Context Protocol (MCP)

**MCP** standardizes how an application exposes **resources**, **prompts**, and **tools** to a model host. When a Neo4j-related process speaks MCP, an agent framework can **discover** capabilities the same way across vendors—instead of writing a one-off HTTP wrapper for every integration.

For GraphAcademy learners, MCP is the same conceptual bridge as in link:/courses/genai-mcp-neo4j-tools/[Developing with Neo4j MCP Tools^]: a **contract** between the model runtime and your data plane.

[.slide]
== Agent-to-agent (A2A) and workflows

**A2A** patterns let one agent **delegate** to another (different instructions, different tools, different safety posture). Combined with orchestration, you can model a **planner** agent that calls a **retrieval** agent and a **verification** agent before answering.

Framework releases also emphasize **streaming** responses, **checkpointing** state for long runs, and **human-in-the-loop** gates when automation must stop for approval—requirements that show up in real deployments even when tutorials stay short.
