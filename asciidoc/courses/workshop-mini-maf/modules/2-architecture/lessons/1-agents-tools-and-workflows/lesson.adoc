[.slide.discrete]
== Introduction

In Microsoft Agent Framework: 

* An **agent** is a program object that combines a **language model**, **instructions**, and **tools**. 
* A **tool** is anything the model can invoke with structured arguments: HTTP APIs, database clients, search, **MCP servers**, or your own functions.
* **Orchestration** is how you connect multiple steps: sequential chains, branches, parallel fan-out, or explicit hand-offs between agents. The framework expresses the orchestration patterns in code and, in many samples, as **graph-shaped** workflows.

[.slide]
== Agents and tools

**Agent:** Holds policy text ("what you are allowed to do"), model configuration, and a **tool registry**. The agent decides *when* to call tools based on user input and prior turns.

**Tool:** A specific, testable function. Good tools return **typed** or **JSON-friendly** results so the model can reason over them. Heavy business logic still belongs in **your** libraries; the tool is the adapter.

The separation of agent and tools mirrors what you learned in link:/courses/genai-fundamentals/[Neo4j & Generative AI Fundamentals^] and related GraphRAG labs: retrievers and Cypher-backed helpers behave like **tools** feeding the agent conversation.

[TIP]
.Tool descriptions
Keep tool descriptions **specific**: state what the tool returns and when the model should pick it over other tools. Vague tools make tool selection unreliable - especially when several tools can touch the same graph.

[.slide]
== Model Context Protocol (MCP)

**MCP** standardizes how an application exposes **resources**, **prompts**, and **tools** to a model host. When a Neo4j-related process speaks MCP, an agent framework can **discover** capabilities the same way across vendors - instead of writing a one-off HTTP wrapper for every integration.

[TIP]
.Learn more about MCP and Neo4j
====
You can learn more about MCP and integration with Neo4j in the GraphAcademy course link:/courses/genai-mcp-neo4j-tools/[Developing with Neo4j MCP Tools^]
====

[.slide]
== Agent-to-agent (A2A) and workflows

**A2A** patterns let one agent **delegate** to another (different instructions, different tools, different safety posture). Combined with orchestration, you can model a **planner** agent that calls a **retrieval** agent and a **verification** agent before answering.

[NOTE]
.Agent-to-Agent (A2A) Protocol
====
The link:https://a2a-protocol.org/latest/[Agent-to-Agent (A2A) Protocol^] is an open protocol developed by Google and describes discovery, task exchange, and streaming between agent runtimes when you need a published contract beyond in-process delegation.
====

The framework also emphasizes **streaming** responses, **checkpointing** state for long runs, and **human-in-the-loop** gates when automation must stop for approval.
